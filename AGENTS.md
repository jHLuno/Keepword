> Этот файл читают AI-кодеры: Codex, Cursor, Claude Code, Devin, Copilot
> Workspace, Aider и другие.
>
> Цель — **экономить токены контекста**, не терять продуктовую логику и не
> превращать Keepword в случайный набор функций.
>
> В репозитории могут быть исторические документы, старые планы, логи и

## 1. TL;DR проекта

- **Keepword** — Telegram-бот, который замечает обещания, поручения и рабочие договорённости в командных чатах и помогает доводить их до результата.
- Главный сценарий: сообщение в группе → AI замечает обязательство → публичный reply → автор или админ подтверждает → личное напоминание → личный вечерний digest.
- Fallback: пользователь пересылает сообщение боту в личный чат, если бот не подключён к группе или команда не хочет авто-анализ.
- MVP — **Telegram-first**: без web-dashboard, CRM, Notion, ClickUp, WhatsApp, чтения старой истории и автосоздания задач без подтверждения.
- Главный принцип: **AI видит много, вмешивается редко**. Бот не должен засорять чат, выдумывать обязательства или создавать задачи без явного подтверждения.
- Главный продуктовый документ — [`PROJECT.md`](./PROJECT.md). Если добавляется функция после MVP или меняется логика продукта, `PROJECT.md` **обновляется обязательно**.

## 2. Главные продуктовые правила

Перед любой реализацией помни:

1. Keepword — не ещё один task manager.
2. Keepword закрывает разрыв между **договорились в чате** и **задача реально зафиксирована**.
3. Бот не должен реагировать на каждое сообщение.
4. Бот не должен создавать commitment без подтверждения автора исходного сообщения или администратора чата.
5. Каждое commitment должно иметь источник: `chat_id`, `message_id`, автора, время и при необходимости контекстную цепочку сообщений.
6. Если AI не уверен — он не делает вид, что уверен.
7. Если неясен срок или исполнитель, бот должен запросить уточнение, а не придумывать значение.
8. Личные напоминания и digest доступны только после того, как пользователь открыл личный чат с ботом и нажал `/start`.
9. Бот анализирует только новые сообщения после подключения к группе. История чата не читается в MVP.
10. Пользователь и администратор должны иметь возможность отключить уведомления и удалить данные, связанные с чатом.

Полная продуктовая логика, MVP-границы и сценарии находятся в [`PROJECT.md`](./PROJECT.md).

## 5. Что делать перед кодом

Для любой задачи сложнее мелкой правки текста или стиля сначала:

1. Прочитай `PROJECT.md` и этот файл.
2. Найди текущую реализацию и связанные тесты.
3. Сформулируй, какой пользовательский сценарий меняется.
4. Определи, какие данные затрагиваются.
5. Проверь риски: privacy, права доступа, дубли задач, spam, удаление данных.
6. Напиши короткий план до начала изменений.

Пример хорошего плана:

```text
План:
1. Добавить таблицу onboarding_tokens.
2. Создать deep link при подключении группы.
3. Обработать /start join_<token> в личном чате.
4. Привязать пользователя к workspace и chat.
5. Добавить onboarding-тесты и проверку истёкшего токена.
6. Обновить PROJECT.md и CHANGELOG.md.
```

Не начинай крупный рефакторинг или изменение схемы базы без плана.

Каждая нетривиальная задача должна пройти этот цикл:

```text

        ↓
Прочитать минимальный контекст
        ↓
Составить короткий план
        ↓
Внести маленькие, обратимые изменения
        ↓

        ↓
Добавить полезные логи
        ↓
Запустить проверки
        ↓

```

### Принцип маленьких вертикальных срезов

Не делай «весь продукт» одним большим изменением.

Правильная последовательность:

```text
Схема данных
        ↓

        ↓
Telegram handler
        ↓

        ↓
Напоминания
        ↓
Digest
        ↓

```

Каждый этап должен оставлять проект в рабочем состоянии.

## 7. Логирование — обязательная часть разработки

> Сначала структура → потом логи → потом сложная логика.

Логи нужны не только для production. Для AI-разработки они являются способом быстро понять, что реально произошло, и не исправлять один и тот же баг циклично.

### Что логировать

Логируй ключевые события с безопасными метаданными:

```text
telegram_update_received
message_candidate_detected
message_skipped_by_pre_filter
llm_commitment_extraction_started
llm_commitment_extraction_completed
llm_commitment_extraction_failed
commitment_suggestion_created
commitment_confirmed
commitment_rejected
commitment_updated
reminder_sent
reminder_delivery_failed
daily_digest_sent
daily_digest_failed
onboarding_link_created
onboarding_completed
authorization_denied
duplicate_commitment_detected
```

### Что должно быть в структурированном логе

Где это применимо, добавляй:

```text
timestamp
level
event_name
request_id
workspace_id
telegram_chat_id
telegram_user_id
message_id
commitment_id
duration_ms
result
error_code
```

### Что запрещено логировать

Никогда не логируй:

- Bot token
- OpenAI API key
- Пароли
- JWT или session tokens
- Полный текст приватных сообщений
- Полный контекст чата
- Личные данные, если достаточно ID
- Содержимое `.env`

Если для дебага нужен фрагмент текста, используй минимально необходимую маскированную версию или хеш.

## 8. AI-логика и правила качества

### 8.1. Как работает извлечение обязательства

LLM получает только потенциально релевантное сообщение и короткий контекст.

Она должна вернуть структурированный результат, а не свободный текст.

Пример ожидаемой структуры:

```ts
type CommitmentCandidate = {
  is_commitment: boolean;
  category: "promise" | "assignment" | "follow_up" | "none";
  title: string | null;
  description: string | null;
  assignee_telegram_user_id: string | null;
  due_at: string | null;
  due_date_text: string | null;
  confidence: "high" | "medium" | "low";
  source_message_ids: string[];
  needs_assignee_clarification: boolean;
  needs_due_date_clarification: boolean;
  reasoning_short: string;
};
```

### 8.2. Правила вмешательства

Публичный reply допустим только если:

- Есть конкретное действие
- Понятен исполнитель или его можно достоверно определить
- Есть срок или бот может кратко запросить его
- Уверенность высокая
- Нет дубликата существующей задачи
- Чат работает в режиме `Suggest`

Если уверенность средняя:

- Не создавай commitment
- Не выдавай догадку за факт
- Можно задать короткий уточняющий вопрос

Если уверенность низкая:

- Ничего не делай
- Не засоряй чат

### 8.3. Запрещённые AI-поведения

AI и код не должны:

- Выдумывать задачу, срок, исполнителя или цитату
- Создавать commitment без подтверждения в MVP
- Считать шутку обещанием
- Отправлять публичные сообщения о личных просрочках исполнителя
- Раскрывать данные другого чата или workspace
- Сохранять весь чат без необходимости
- Отправлять каждый входящий message в LLM
- Писать в группу без продуктовой причины

## 9. Privacy, доступы и безопасность

Рабочие переписки могут содержать чувствительную информацию.

### Обязательные правила

- Доступ к данным строго изолирован по `workspace` и `chat`.
- Админ чата может менять настройки чата и просматривать командную сводку.
- Автор сообщения или админ подтверждает и редактирует созданный из сообщения commitment.
- Участник видит в личном digest только свои задачи.
- Не доверяй Telegram ID, chat ID, callback data или deep-link payload без серверной проверки.
- Deep-link onboarding token должен быть одноразовым или подписанным, иметь срок действия и быть связан с конкретным чатом.
- Не добавляй настоящие `chat_id` в публичный URL deep link.
- При удалении чата необходимо удалить или обезличить связанные сообщения, контекст, commitments, reminders и onboarding tokens в соответствии с выбранной политикой retention.
- Все секреты должны быть только в environment variables.
- `.env` и ключи никогда не коммитятся.

### Telegram-специфика

- Бот не может первым начать личный чат с пользователем.
- Для личных напоминаний пользователь обязан нажать `/start` через onboarding deep link.
- Если пользователь не подключил личные уведомления, не раскрывай его личную просрочку в общем чате.
- Если бот анализирует новые сообщения группы, команда должна видеть onboarding-сообщение с объяснением того, какие данные и зачем обрабатываются.

## 10. Product documentation всегда актуальна

`PROJECT.md` — не одноразовый документ для MVP. Это текущий источник правды о продукте.

### Когда обязательно обновлять `PROJECT.md`

Обновляй `PROJECT.md`, если:

- Добавляется новая пользовательская функция
- Меняется основной сценарий
- Меняется MVP-граница
- Добавляется новый режим работы чата
- Добавляется интеграция: Notion, Google Calendar, Google Sheets, ClickUp, WhatsApp, email и т.д.
- Меняется модель ролей, прав или уведомлений
- Добавляется новый AI workflow
- Меняется правило хранения или удаления данных
- Меняется технический поток, который важен для работы продукта
- Решение из `OPEN_QUESTIONS.md` становится утверждённым

### Как обновлять

Не переписывай весь файл без причины.

1. Найди релевантный раздел.
2. Обнови сценарий, MVP, ограничения и техническую логику.
3. Если функция больше не MVP — перенеси её в раздел «Будущее проекта».
4. Добавь заметку в `CHANGELOG.md`.
5. Укажи в `HANDOFF.md`, что продуктовая документация обновлена.

**Ни одна значимая функция после MVP не считается завершённой, пока `PROJECT.md` не отражает её актуальную логику.**

## 11. Файлы состояния сессии

После каждой нетривиальной сессии обновляй:

1. **`CHANGELOG.md`** — что сделано и проверено.
2. **`HANDOFF.md`** — что сделано, что не сделано, что блокирует продолжение.
3. **`OPEN_QUESTIONS.md`** — новые продуктовые/технические вопросы или закрытые решения.
4. **`PROJECT.md`** — только если изменились продукт, MVP, UX-сценарий, права, privacy или архитектура.

### Формат `CHANGELOG.md`

```markdown
## YYYY-MM-DD — Короткое название задачи

### Added
- Что добавлено

### Changed
- Что изменено

### Fixed
- Что исправлено

### Verified
- Какие команды, тесты или ручные сценарии пройдены

### Notes
- Ограничения или известные риски
```

### Формат `HANDOFF.md`

```markdown
## YYYY-MM-DD — Handoff

### Done
- Что готово

### Not done
- Что сознательно не делалось

### Risks / blockers
- Что блокирует или требует решения владельца

### Next recommended step
- Самый логичный следующий шаг
```

### Архивирование

Если в `CHANGELOG.md` или `HANDOFF.md` стало больше трёх сессионных записей:

- Перенеси все записи, кроме двух последних, в `docs/archive/`.
- Например: `docs/archive/CHANGELOG-pre-S04.md`.
- В корневом файле оставь короткую ссылку на архив.

Цель — сохранить активный контекст маленьким для следующего AI-агента.

## 13. Минимальные тестовые сценарии

При работе с Keepword всегда думай о следующих сценариях:

### Обязательства

- Высокая уверенность: «Я отправлю КП сегодня»
- Обязательство с контекстом: «Я» → «До завтра»
- Средняя уверенность: «Давайте вернёмся к этому в пятницу»
- Низкая уверенность: «Надо бы посмотреть»
- Дубликат обязательства
- Обещание без срока
- Обещание без понятного исполнителя
- Шутка, которая содержит триггерное слово
- Отмена или редактирование предложения

### Права

- Автор сообщения подтверждает задачу
- Администратор подтверждает задачу
- Обычный участник не может подтвердить чужую задачу
- Пользователь из другого чата не видит данные
- Удалённый администратор теряет права

### Onboarding

- Валидный deep link подключает уведомления
- Просроченный token отклоняется
- Использованный одноразовый token отклоняется
- Token другого чата не создаёт доступ
- Пользователь без `/start` не получает личные уведомления
- Повторное приглашение не спамит чаще установленного лимита

### Напоминания и digest

- Reminder уходит только исполнителю
- Просрочка не публикуется в общий чат
- Личный digest содержит только задачи получателя
- Админский digest содержит агрегированные риски команды
- Повторный cron не отправляет один digest дважды
- Ошибка Telegram delivery логируется и безопасно обрабатывается

## 15. Бюджет контекста

Ориентир для одной сессии:

- До **30K токенов**: `AGENTS.md` + `PROJECT.md` + README + последние записи CHANGELOG/HANDOFF.
- До **5K–10K токенов** дополнительно: конкретные файлы, которые нужно менять.
- Если файл больше примерно 50 KB — не читай его полностью, пока не доказано, что это необходимо.
- Если собираешься открыть целиком `CHANGELOG.md`, `HANDOFF.md`, большой лог, архив или план — остановись и используй поиск.

Контекст — ограниченный ресурс.  
Читайте ровно столько, сколько нужно, чтобы безопасно решить текущую задачу.

## 17. Если задача неясна

Если вопрос влияет на продукт, privacy, права, хранение данных, стоимость, интеграции или основной пользовательский сценарий:

1. Не выдумывай решение.
2. Опиши 2–3 понятных варианта.

4. Добавь вопрос в `OPEN_QUESTIONS.md`.
5. Попроси владельца проекта выбрать вариант.

Если деталь маленькая и не меняет продуктовую логику — выбери разумное решение, зафиксируй предположение в коде или handoff и двигайся дальше.
